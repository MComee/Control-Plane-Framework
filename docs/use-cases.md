# Use Cases

## Private application development

Use the framework as the public control-plane reference and fork it into a private repository for product-specific planning, implementation, and validation.

## Internal tooling

Use the framework to define mission, constraints, execution tasks, and acceptance criteria before building an internal utility or operations tool.

## CLI tool development

Use the framework when a project will involve command-line AI tools, local iteration, and structured review before accepting changes.

## Research or experimentation

Use the framework when a project needs a reusable structure for planning, documenting assumptions, recording constraints, and preserving review discipline during exploratory work.

## Multi-model workflows

Use the framework when one project may use multiple chat-based tools and multiple command-line tools, with tool choice varying by task rather than being fixed globally.

## When not to use this framework

This framework is likely unnecessary when:

- the project is extremely small and short-lived
- no repository-level governance is needed
- planning and validation overhead would outweigh the value of structure
- the work is intentionally informal or disposable
