# Start Here

If you are new to Control Plane Framework, use this sequence:

1. Read the top-level `README.md` for the repository purpose.
2. Read `docs/overview.md` for the operating model.
3. Read `docs/architecture.md` for the layer and lane structure.
4. Read `docs/tool-selection.md` for the multi-tool, model-agnostic workflow stance.
5. Read `docs/repository-governance.md` if you want to keep `main` clean and reviewed.
6. Read `docs/forking-pattern.md` before starting a private project from the framework.
7. Review `examples/sample-project-brief.md` to see how a new project can begin.

## Fast path

If your goal is to start a new project quickly:

- fork the repository
- complete the planning templates
- define governance constraints
- set acceptance criteria
- begin implementation only after the basic control structure is in place

## Project initialization requirement

Before real implementation work begins in a derived project, the project operating structure must be instantiated.

At minimum, this includes:

- a canonical project workspace
- a protected core vision
- initial module decomposition
- initial task artifacts
- priority files
- an active-work handoff package

See:

- [Project Operating Contract](/docs/project-operating-contract.md)
- [Project Initialization Standard](/docs/project-initialization-standard.md)
- [Active Work Handoff](/docs/active-work-handoff.md)
- [Planning Synchronization Rule](/docs/planning-synchronization-rule.md)
